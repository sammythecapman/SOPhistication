# Replit Agent Prompt: Tiered Confidence Scoring + Cumulative Learning for SBA Extraction Pipeline

## Context

I have an SBA loan document extraction application that uses a two-layer NLP pipeline:

1. **spaCy NER preprocessing**: Extracts named entities (persons, organizations, money, dates, etc.) from raw PDF text and passes them as "hints" to the LLM.
2. **Claude API extraction**: Takes the raw text + NER hints and extracts structured fields (borrower name, lender name, loan amount, guarantors, etc.) into JSON.
3. **Post-extraction validation**: Cross-references Claude's extracted values against the spaCy NER entity set and flags mismatches as "possible hallucination."

### The Problem

The current validation step is binary — if an extracted value doesn't appear in the spaCy NER entity set, it gets flagged with a generic warning: "Field 'X' value 'Y' not found in NER entity set — verify this is correct (possible hallucination)."

This creates a **"cry wolf" problem**. Most flags are false positives caused by spaCy's NER model failing to detect legitimate entities (e.g., bank names, names with honorifics like "Dr.", less common person names), NOT by Claude hallucinating. Reviewers see warnings on nearly every extraction and start ignoring them, which defeats the purpose of the validation layer entirely.

## What I Need Built

Two interconnected features:

### Feature 1: Tiered Confidence Scoring System

Replace the binary "found/not found" validation with a three-tier confidence system:

**Tier 1 — RED (High Risk: Likely Hallucination)**
- The extracted value does NOT appear anywhere in the raw source text — not as an NER entity, not as a substring, not as a fuzzy match.
- This is the ONLY scenario where "possible hallucination" is appropriate.
- Display: Red warning banner with strong language. This should feel urgent.
- Example message: "⛔ HALLUCINATION RISK: Field 'PersonalGuarantor2' value 'James Mitchell' was NOT FOUND anywhere in the source document text. This value may have been fabricated by the LLM. Manual verification against the original document is required."

**Tier 2 — YELLOW (Low Risk: NER Coverage Gap)**
- The extracted value DOES appear in the raw source text (confirmed via case-insensitive substring match OR fuzzy match above a similarity threshold of ~85%), but was NOT tagged as an entity by spaCy's NER model.
- This means the value is real — it's in the document — spaCy just didn't recognize it as a named entity. Low hallucination risk.
- Display: Subtle yellow/amber advisory. Informational, not alarming.
- Example message: "⚠️ NER gap: Field 'LenderName' value 'Community Trust Bank' was found in source text but not detected by spaCy NER. Likely a model coverage gap — low hallucination risk."
- Include a brief source text snippet showing where the value was found (±50 characters of surrounding context) so the reviewer can confirm at a glance.

**Tier 3 — GREEN (Confirmed: NER Validated)**
- The extracted value matches an entity in the spaCy NER set (exact or fuzzy match).
- No warning displayed to the user. Optionally show a small green checkmark or "NER confirmed" badge in a detailed view.

**Implementation details for the matching logic:**

- **Exact match**: `extracted_value.lower().strip() in [ent.lower().strip() for ent in ner_entities]`
- **Fuzzy match against NER set**: Use `difflib.SequenceMatcher` or `rapidfuzz` (preferred if available) with a threshold of 85% similarity. This catches cases like spaCy detecting "Rebecca Thornton" while Claude extracted "Dr. Rebecca Thornton."
- **Source text presence check**: For values not in NER set, check `extracted_value.lower() in raw_source_text.lower()`. If that fails, try fuzzy substring matching — the value might have minor formatting differences from how it appears in the PDF text.
- **Field-type awareness**: Different fields should be validated against appropriate NER categories:
  - Person fields (BorrowerName, PersonalGuarantor1-3, SellerName, etc.) → check against PER entities
  - Organization fields (LenderName, BorrowerEntityName, etc.) → check against ORG entities
  - Money fields (LoanAmount, SBAGuarantyAmount, etc.) → check against MONEY entities
  - Date fields → check against DATE entities
  - For fields that could be either a person or org (like BorrowerName), check against both

**Output**: Each extracted field should carry a confidence object:

```json
{
  "LenderName": {
    "value": "Community Trust Bank",
    "confidence_tier": "yellow",
    "ner_match": false,
    "source_text_match": true,
    "source_snippet": "...the Lender, Community Trust Bank, a federally chartered...",
    "match_details": "Exact substring match in source text at character position 1247"
  }
}
```

### Feature 2: Cumulative Learning System (Feedback Loop)

Build a persistent feedback mechanism that logs reviewer decisions on flagged fields and uses that history to improve future confidence scoring.

**Feedback capture:**
- For every RED or YELLOW flagged field displayed to the reviewer, show two buttons: "✅ Correct" and "❌ Incorrect"
- "Correct" means the extracted value was accurate despite the flag (false positive warning)
- "Incorrect" means the extracted value was wrong and the flag was justified (true positive warning)
- Store each feedback event with:
  - Timestamp
  - Field name (e.g., "LenderName")
  - Extracted value
  - Confidence tier at time of flag (red or yellow)
  - Reviewer verdict (correct or incorrect)
  - NER entities that were available for that field type (for debugging)
  - Source document identifier (filename or hash)

**Persistent storage:**
- Store feedback in a local JSON file (`validation_feedback_log.json`) or SQLite database — whichever fits the app architecture better.
- The log should be append-only and human-readable.

**Learning layer (threshold adjustment):**
- Track per-field-type accuracy rates. For example: "LenderName yellow flags have been marked 'correct' 95% of the time across 40 reviews."
- When a field type's yellow-flag false positive rate exceeds 90% over a minimum of 20 reviews, automatically promote that field type's yellow warnings to green (suppress them). Display a subtle note in a settings/admin panel: "LenderName warnings auto-suppressed — 95% false positive rate over 40 reviews."
- When a field type's red-flag false positive rate exceeds 80% over a minimum of 10 reviews, downgrade future reds to yellows for that field type (but never suppress reds entirely).
- These threshold adjustments should be reversible — an admin toggle to reset learning for a field type.
- On startup, load the feedback log and compute current accuracy rates to apply any learned suppressions.

**Dashboard / analytics view:**
- A simple page or panel that shows:
  - Total extractions processed
  - Total flags generated (broken down by red/yellow)
  - Total flags reviewed (with reviewer verdict breakdown)
  - Per-field-type false positive rates
  - Which field types have been auto-suppressed
  - A table of the most recent 50 feedback events

## Technical Constraints

- The extraction pipeline is written in Python. The application frontend can be whatever framework fits best (the current version uses a web interface).
- spaCy (`en_core_web_sm`) is already installed and used for NER.
- The Anthropic API (Claude) is already integrated for extraction.
- For fuzzy matching, prefer `rapidfuzz` if adding a dependency is acceptable, otherwise use `difflib.SequenceMatcher` from the standard library.
- The tiered scoring logic should be modular — a standalone function or class that takes (extracted_fields, ner_entities, raw_source_text) and returns the confidence-annotated fields.
- The feedback/learning system should be decoupled enough that it can be disabled without breaking extraction.

## What Success Looks Like

1. A reviewer processing an SBA document sees **zero or near-zero warnings** on clean extractions where spaCy simply missed a bank name or guarantor name (these become green or low-priority yellow).
2. When a genuine hallucination occurs (Claude fabricates a value not in the document), the reviewer sees a **red flag that feels urgent and trustworthy** because they aren't drowning in false alarms.
3. Over time, as reviewers provide feedback, the system **learns which field types produce false positives** and automatically adjusts, further reducing noise.
4. The feedback dashboard gives visibility into system performance and makes the learning process transparent.

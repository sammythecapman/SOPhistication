# Replit Prompt: SBA Loan Data Extraction Tool with SharePoint Integration

## What This Project Is

I have a working Python script (attached below or in this repo as `sba_extract_standalone.py`) that extracts structured data from SBA loan documents (Terms & Conditions PDFs and optional Credit Memo PDFs). I want to turn it into a full-stack web application with a SharePoint integration so my law firm can use it as an internal tool.

## How the Existing Script Works

The script has a 3-stage pipeline:

1. **PDF Ingestion**: Reads PDF text using `pdfplumber`.
2. **NER Preprocessing**: Runs spaCy's `en_core_web_sm` Named Entity Recognition model over the extracted text. Tags PER, ORG, LOC, GPE, MONEY, DATE, and PERCENT entities. Deduplicates and formats them into a structured hint block.
3. **LLM Extraction with NER Hints**: Sends the full document text + NER hint block to the Anthropic Claude API (`claude-sonnet-4-20250514`). Claude receives a dynamically-built JSON schema based on deal structure (asset purchase vs. stock purchase vs. construction, number of guarantors, etc.) and returns populated JSON.
4. **Post-Extraction Validation**: Cross-references Claude's extracted names/orgs against the NER entity set and flags mismatches as potential hallucinations.
5. **Formatting**: Applies currency formatting ($X,XXX,XXX.XX), percentage formatting (X.XX%), date formatting (Month DD, YYYY), and number-to-words conversion for long-format fields. Also runs regex fallback extraction for critical fields like LoanNumber and MaturityDate that Claude sometimes misses.

The script outputs two JSON files: a complete extraction report and a clean formatted-data-only file.

### Key Functions in the Script
- `load_ner_model()` — loads spaCy NER model
- `run_ner()` — runs NER over document text, returns entities grouped by label
- `merge_ner_results()` — merges NER from Terms & Credit Memo
- `format_ner_hints()` — formats NER entities into a prompt hint block
- `validate_extraction_against_ner()` — post-extraction validation
- `regex_extract_critical_fields()` — regex fallback for LoanNumber, MaturityDate
- `read_pdf()` — extracts text from PDF via pdfplumber
- `analyze_deal_structure()` — first Claude API call to determine deal type, guarantor count, etc.
- `build_schema()` — dynamically builds extraction schema based on deal structure
- `extract_fields()` — second Claude API call with NER hints injected into prompt
- `apply_field_formatting()` — formats all extracted values (currency, dates, percentages, etc.)
- Formatting helpers: `format_currency_short()`, `format_percentage_short()`, `format_date_long()`, `percentage_to_words()`, `number_to_words()`, `infer_state_from_address()`

### Current Dependencies
- `anthropic` (Claude API client)
- `pdfplumber` (PDF text extraction)
- `spacy` + `en_core_web_sm` model (NER preprocessing)
- `python-dateutil` (optional, for MaturityDate calculation)

## What I Want You to Build

### 1. Web Application (Flask or FastAPI backend, React or simple HTML/JS frontend)

**Frontend UI should include:**
- A clean, professional interface appropriate for a law firm
- File upload area for PDF(s) — drag-and-drop preferred. User uploads a Terms & Conditions PDF and optionally a Credit Memo PDF
- A "Run Extraction" button
- Real-time progress indicators showing each pipeline stage (Reading PDF → Running NER → Analyzing Deal → Extracting Fields → Validating → Formatting)
- Results display: show the formatted extracted data in a clean table/card layout, color-coded (green for populated fields, gray for empty)
- NER validation warnings displayed prominently if any exist
- "Download JSON" button for the formatted output
- "Push to SharePoint" button (see below)
- Extraction history / past runs visible in a sidebar or separate tab

**Backend should:**
- Accept PDF uploads via API endpoint
- Run the full existing pipeline (NER preprocessing → deal analysis → extraction → validation → formatting)
- Store extraction results (SQLite is fine for now)
- Expose API endpoints for SharePoint integration
- Handle errors gracefully and return meaningful status messages

### 2. SharePoint Integration

This is critical. The firm stores loan documents in SharePoint and the extracted data needs to flow back into SharePoint. Build the following:

**Reading from SharePoint:**
- Authenticate to SharePoint Online using Microsoft Graph API (OAuth 2.0 with MSAL)
- User should be able to browse/select a SharePoint document library folder
- Pull PDF files directly from SharePoint instead of manual upload
- Support selecting a Terms & Conditions PDF and a Credit Memo PDF from SharePoint

**Writing to SharePoint:**
- After extraction, push the formatted JSON data to a SharePoint list (one row per deal, columns matching the extraction schema fields)
- OR write the JSON output file back to a specified SharePoint document library folder
- Support both options — let the user choose "Save to SharePoint List" or "Save to SharePoint Folder"

**SharePoint Auth Flow:**
- Use Microsoft Authentication Library (MSAL) for Python
- Support the authorization code flow for user-delegated permissions
- Store tokens securely and handle refresh
- Required Microsoft Graph permissions: `Sites.ReadWrite.All`, `Files.ReadWrite.All`
- The app will need to be registered in Azure AD — provide instructions or a setup script for this

**SharePoint Configuration:**
- Let the user configure their SharePoint site URL, document library name, and target list name via environment variables or a settings page in the UI
- Environment variables needed: `SHAREPOINT_SITE_URL`, `SHAREPOINT_CLIENT_ID`, `SHAREPOINT_CLIENT_SECRET`, `SHAREPOINT_TENANT_ID`

### 5. Important Implementation Notes

- The existing script's `API_KEY` is hardcoded — replace this with `os.environ.get("ANTHROPIC_API_KEY")` in the web version
- The spaCy model needs to be downloaded at build time. Add `python -m spacy download en_core_web_sm` to the Replit build/setup step
- The extraction pipeline makes 2 Claude API calls per document (deal analysis + field extraction). These take 5-15 seconds each, so the UI needs async handling or websocket progress updates
- Keep all the existing extraction logic intact — it works. The web app is a wrapper around the existing pipeline, not a rewrite
- The NER hint injection into the Claude prompt is the core innovation — don't simplify or remove it

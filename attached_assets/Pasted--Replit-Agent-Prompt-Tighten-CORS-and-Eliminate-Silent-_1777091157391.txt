# Replit Agent Prompt — Tighten CORS and Eliminate Silent Failures

## Goal

Two related production-readiness fixes in `artifacts/sba-backend/`:

1. Replace the wide-open `CORS(app, origins="*")` policy with an explicit allowlist driven by an environment variable.
2. Eliminate silent failure modes in the extraction pipeline so that a failed Claude call, a malformed JSON response, or a degraded extraction is logged, surfaced in the API response, and visible in the UI — never swallowed into an empty dict that looks like a normal "field not found" result.

Do not touch the React frontend logic, the database schema, or the prompts themselves. This is a backend hardening pass only.

---

## Part A — CORS

**Current state:** `artifacts/sba-backend/app.py` line 29 reads:

```python
CORS(app, origins="*")
```

This means any origin can call `/api/extract`, `/api/extractions`, `/api/extractions/<id>/download`, etc. That undercuts the Fernet-at-rest and signed-token story already in place.

**Required change:**

1. Add a new environment variable `ALLOWED_ORIGINS` — a comma-separated list of allowed origins (e.g. `https://jbl-sba-data-extractor.smcapper4.repl.co,http://localhost:5173`).
2. Parse it at app startup. Strip whitespace, drop empty entries.
3. If `ALLOWED_ORIGINS` is unset OR empty, fall back to `http://localhost:5173,http://localhost:21230` (dev defaults — the Vite frontend port and the proxied port from `replit.md`). Do **not** silently fall back to `*`.
4. Replace the existing `CORS(app, origins="*")` call with `CORS(app, origins=<parsed_list>, supports_credentials=True)`.
5. Log the resolved allowlist at startup (`app.logger.info(f"CORS allowed origins: {origins}")`) so it's visible in Replit's logs.
6. Update `replit.md` under "Environment Variables Required" to document `ALLOWED_ORIGINS` (mark it Required for production, Optional for dev with the documented fallback).

**Acceptance criteria:**

- A `curl -H "Origin: https://evil.example.com" -X OPTIONS https://<backend>/api/extract` preflight returns no `Access-Control-Allow-Origin` header (or a header that does not match the evil origin).
- A request from the configured frontend origin still works end-to-end (upload → poll → results).
- Setting `ALLOWED_ORIGINS=""` or unsetting it does NOT result in `*`. It results in the localhost dev defaults.
- `replit.md` reflects the new env var.

## Part B — No more silent failures

**Current state:** Two functions in `artifacts/sba-backend/extraction/schemas.py` each end with a bare `except Exception: return {}`:

- `analyze_deal_structure` (line ~80) — when Claude's deal-analysis JSON fails to parse, returns `{}`. The downstream `build_schema({})` then produces a stripped-down schema with no second borrower, no guarantors, no real estate, no construction, no seller. The user sees a sparse result and has no idea the deal-analysis call failed.
- `extract_fields` (line ~269) — when the field-extraction JSON fails to parse, returns `{}`. The pipeline then runs only the regex fallbacks (which only cover `LoanNumber` and `MaturityDate` per `regex_fallbacks.py`), so the user gets a result with two populated fields and 23+ blank ones — indistinguishable from a document where Claude just couldn't find the data.

This must change. Blanks must mean "not in the document," not "the AI call silently exploded."

**Required changes:**

### B1. Replace bare excepts with explicit, logged failure handling.

In both `analyze_deal_structure` and `extract_fields`:

- Catch `json.JSONDecodeError` separately from other exceptions.
- On `json.JSONDecodeError`: log at `ERROR` level with `app.logger` (or `logging.getLogger(__name__)`), include the first 500 chars of the raw Claude response, the function name, and a short error message. Then raise a new custom exception `ExtractionStageError` (define it in `extraction/__init__.py` or a new `extraction/errors.py`) with a structured payload: `{"stage": "deal_analysis" | "field_extraction", "reason": "json_decode", "raw_excerpt": "..."}`.
- On other exceptions (anthropic API errors that escaped the retry wrapper, network errors, etc.): log and re-raise as `ExtractionStageError` with `reason="api_error"` and the original exception message.
- Do NOT return `{}` from these functions anymore. Either return a valid dict or raise `ExtractionStageError`.

### B2. Catch `ExtractionStageError` at the pipeline level.

In `extraction/pipeline.py`, `run_extraction_pipeline`:

- Wrap the `analyze_deal_structure` call. If it raises `ExtractionStageError`, do NOT crash the whole job. Instead, set a `degraded` flag on the result and continue with an empty `deal` dict (current behavior) — but record the degradation reason.
- Wrap the `extract_fields` call the same way. If it raises, again record the degradation, set `raw_data = {}`, and let the regex fallbacks run. The result must include the degradation info.

The result dict returned by `run_extraction_pipeline` should grow a new top-level key:

```python
"extraction_health": {
    "degraded": True | False,
    "stage_failures": [
        {"stage": "deal_analysis", "reason": "json_decode", "message": "..."},
        # ...
    ]
}
```

### B3. Surface the health info through the API.

In `app.py`, the `/api/jobs/<job_id>` endpoint and the `/api/extractions/<id>` endpoint must include `extraction_health` in their JSON responses. Persist it to the `sba_extractions` table — add a nullable `extraction_health` JSONB column via a migration in `db.py` (follow whatever pattern the existing CRUD code uses; if there's no migration system, add the column idempotently in the same `init_db`/equivalent function that creates the tables, using `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`).

### B4. Surface the health info in the UI.

In `artifacts/sba-web/src/pages/Results.tsx` (and `ExtractionView.tsx` if it shows the same data), add a visible warning banner at the top of the results when `extraction_health.degraded === true`. It should:

- Use a yellow/amber alert style (not red — extraction still partially completed).
- List the failed stages in plain English: "Deal structure analysis failed (malformed AI response). Some fields may be missing because the schema was incomplete."
- Be dismissible per-result but not hidden by default.

Reuse whichever shadcn/ui Alert component is already in the project (it's in `components/ui/`).

### B5. One small bonus while you're in there.

The two retry helpers in `schemas.py` (`_claude_with_retry`) currently `print(...)` on retry. Switch those to `logging.getLogger(__name__).warning(...)` so they show up in structured logs alongside the new error logs.
